## 4. The SCOS_ Prefix — What It Is

Mat's screenshot shows a **different system** (likely SCOS or another VPI service) that uses the `SCOS_SAP_*` naming convention:

| SCOS Name | Equivalent |
|-----------|------------|
| `SCOS_SAP_HOST` | `SAP_HOST` |
| `SCOS_SAP_PORT` | `SAP_PORT` |
| `SCOS_SAP_DB` | `SAP_DB` |
| `SCOS_SAP_USER` | `SAP_USER` |
| `SCOS_SAP_PASS` | `SAP_PASS` |
| `SCOS_SAP_TYPE` | Not used by VieForce HQ |

**This is NOT a current problem** — the `vieforce-hq-api` Cloud Run service has the correct unprefixed names. However, if the service ever migrates to the SCOS naming convention or shares env with an SCOS service, the code would break.

---

## 5. Fix Applied

**Preventive fix in `api/_db.js`** — dual-name fallback:

```javascript
// Before:
server:   process.env.SAP_HOST,
port:     parseInt(process.env.SAP_PORT) || 4444,
database: process.env.SAP_DB,
user:     process.env.SAP_USER,
password: process.env.SAP_PASS,

// After:
server:   process.env.SAP_HOST || process.env.SCOS_SAP_HOST,
port:     parseInt(process.env.SAP_PORT || process.env.SCOS_SAP_PORT) || 4444,
database: process.env.SAP_DB || process.env.SCOS_SAP_DB,
user:     process.env.SAP_USER || process.env.SCOS_SAP_USER,
password: process.env.SAP_PASS || process.env.SCOS_SAP_PASS,
```

Code now works with **either** naming convention. Zero-risk change — the `||` fallback only kicks in if the primary name is undefined.
